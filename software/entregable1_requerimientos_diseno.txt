RESUMEN EJECUTIVO
================================================================

Las instituciones educativas en el Perú registran la asistencia
de manera manual mediante listas de papel o firmas, un proceso
lento, propenso a errores, suplantaciones de identidad y pérdida
de información. Esta situación afecta tanto a docentes como a
estudiantes, y genera una carga administrativa innecesaria.

QhawAI es un sistema inteligente de control de asistencia que
utiliza reconocimiento facial basado en inteligencia artificial
para identificar automáticamente a estudiantes y docentes al
momento de ingresar a la institución. El nombre proviene del
quechua "qhaw", que significa "ver" o "mirar", reflejando la
naturaleza visual del sistema.

El sistema está dirigido a tres tipos de usuarios:
administradores institucionales, operadores (docentes o personal
encargado del registro), y padres de familia. Los
administradores gestionan la configuración del sistema y acceden
a reportes completos. Los operadores usan la cámara web o el
dispositivo móvil para capturar y registrar asistencias en
tiempo real. Los padres reciben notificaciones automáticas
cuando su hijo registra asistencia o llega tarde.

La solución integra un backend web desarrollado en Laravel 13.5,
una API de reconocimiento facial en Python con los modelos
YOLO-Face e InsightFace (ArcFace), una base de datos MySQL y
una aplicación móvil Android. El sistema soporta modo offline
con sincronización automática, detección de tardanza
configurable por turno, generación de reportes en PDF y Excel,
y almacenamiento de evidencia fotográfica de cada registro.

SECCIÓN 1: ESPECIFICACIÓN DE REQUERIMIENTOS
================================================================

1.1 Requerimientos Funcionales
----------------------------------------------------------------

RF-01 — Registro de asistencia por reconocimiento facial
El sistema debe capturar imágenes desde la cámara web o móvil,
procesarlas mediante el modelo de reconocimiento facial y
registrar automáticamente la asistencia de las personas
identificadas, almacenando fecha, hora, turno y nivel de
confianza.

RF-02 — Detección de tardanza
El sistema debe comparar la hora de llegada con el horario de
entrada configurado por turno (mañana/tarde) más la tolerancia
en minutos, marcando el registro como tardanza y calculando
los minutos de retraso.

RF-03 — Enrollment (registro de rostros)
El sistema debe permitir registrar nuevos usuarios capturando
múltiples imágenes de su rostro para construir su perfil
biométrico en la base de datos del modelo de IA.

RF-04 — Modo offline con sincronización
La aplicación móvil debe registrar asistencias localmente
cuando no hay conexión a internet y sincronizarlas
automáticamente con el servidor al recuperar la conexión.

RF-05 — Notificaciones a padres
El sistema debe enviar notificaciones push (FCM) a los padres
de familia registrados cuando su hijo registra asistencia,
indicando si llegó puntual o tarde y la hora exacta.

RF-06 — Gestión de usuarios y roles
El sistema debe gestionar tres roles: administrador (acceso
total), operador (toma de asistencia) y padre de familia
(consulta de asistencia de sus hijos vinculados).

RF-07 — Reportes de asistencia
El sistema debe generar reportes de asistencia filtrados por
día, persona o rango de fechas, mostrando asistencias y faltas,
exportables en formato PDF y Excel.

RF-08 — Historial de asistencia
El sistema debe mostrar el historial de asistencias con foto
del rostro capturado, nombre, grado, hora, turno y estado de
puntualidad, con filtros por nombre, tipo, grado, turno y fecha.

RF-09 — Gestión de grados y secciones
El sistema debe permitir crear, editar y eliminar grados,
secciones y áreas, con configuración individual de horario de
entrada y tolerancia de tardanza por grado.

RF-10 — Cambio de cámara en dispositivos móviles
La aplicación debe detectar si el dispositivo tiene más de una
cámara y permitir al operador cambiar entre cámara frontal y
trasera durante la toma de asistencia.

RF-11 — Registro de ubicación GPS
El sistema debe registrar las coordenadas GPS del dispositivo
al momento de cada registro de asistencia.

RF-12 — Evidencia fotográfica
El sistema debe guardar automáticamente el recorte del rostro
reconocido y el frame panorámico de la cámara (en baja
resolución) como evidencia de cada registro de asistencia.

RF-13 — Panel de administración
El sistema debe proveer un panel web para que el administrador
gestione personas, usuarios, grados, configuración de horarios
y visualice reportes de actividad de operadores.

RF-14 — Sincronización de configuración con la API Python
El sistema debe sincronizar la configuración de turnos (hora de
entrada, tolerancia, corte) desde el panel web hacia la API de
reconocimiento facial automáticamente al guardar cambios.


1.2 Requerimientos No Funcionales
----------------------------------------------------------------

RNF-01 — Rendimiento
El sistema debe procesar y responder a una solicitud de
reconocimiento facial en menos de 2 segundos bajo condiciones
normales de red.

RNF-02 — Precisión del reconocimiento
El modelo de reconocimiento facial debe alcanzar una precisión
mínima del 90% en condiciones de iluminación adecuada, con un
umbral de confianza configurable.

RNF-03 — Disponibilidad
El sistema web debe estar disponible el 95% del tiempo durante
el horario escolar (lunes a viernes, 6:00 a 14:00).

RNF-04 — Seguridad
Todas las rutas del sistema deben requerir autenticación. La
comunicación entre componentes debe realizarse mediante API Key.
Las contraseñas se almacenan con hash bcrypt.

RNF-05 — Escalabilidad
El sistema debe soportar hasta 1000 personas registradas sin
degradación significativa del rendimiento de reconocimiento.

RNF-06 — Usabilidad
La interfaz de toma de asistencia debe ser operable por un
usuario sin capacitación técnica en menos de 5 minutos de uso.

RNF-07 — Compatibilidad móvil
La aplicación Android debe ser compatible con dispositivos que
ejecuten Android 8.0 (API nivel 26) o superior.

RNF-08 — Almacenamiento eficiente
Las fotos panorámicas de evidencia deben almacenarse a
resolución máxima de 640px de ancho con calidad JPEG del 45%,
y los recortes de rostro a calidad 88%.

RNF-09 — Privacidad de datos biométricos
Los datos biométricos (embeddings faciales) se almacenan
únicamente en el servidor de la institución y no se comparten
con servicios externos.

RNF-10 — Interfaz responsiva
La interfaz web debe adaptarse correctamente a resoluciones de
escritorio (1024px+) y dispositivos móviles (320px+).


1.3 Reglas de Negocio
----------------------------------------------------------------

RN-01 — Una persona solo puede tener un registro de asistencia
por día. Los intentos duplicados se ignoran.

RN-02 — El sistema no registra asistencia fuera del horario de
corte configurado por turno. El intento se marca como
"fuera de hora".

RN-03 — Un registro de asistencia offline no puede sobrescribir
un registro previo del mismo día. El primero en llegar tiene
prioridad.

RN-04 — Las notificaciones a padres solo se envían para
estudiantes, no para docentes.

RN-05 — El porcentaje de confianza mínimo para registrar
asistencia es configurable. Por defecto es 55%.

RN-06 — Un día es considerado "día escolar válido" para el
cálculo de faltas si asistió al menos el 10% del total de
personas registradas.

RN-07 — Los reportes de faltas incluyen a todas las personas
(estudiantes y docentes) que no tienen registro de asistencia
en un día escolar válido.


1.4 Restricciones del Sistema
----------------------------------------------------------------

- El reconocimiento facial requiere conexión al servidor Python.
  Sin este servidor activo, la toma de asistencia no es posible
  desde la cámara web.

- El modo offline del app móvil no calcula confianza facial.
  Los registros offline tienen confianza 0%.

- El servidor Python debe ejecutarse en un entorno con GPU o
  CPU compatible con los modelos InsightFace (ONNX).

- La API de Python debe estar en la misma red o accesible vía
  HTTPS para evitar errores de contenido mixto (Mixed Content).

- Los datos biométricos no son portables entre instalaciones
  distintas del sistema.


1.5 Historias de Usuario
----------------------------------------------------------------

HU-01 — Toma de asistencia (Operador)
Como operador, quiero apuntar la cámara hacia los alumnos que
ingresan para que el sistema registre automáticamente su
asistencia sin que tengan que hacer nada.

Criterios de aceptación:
- El sistema detecta el rostro en menos de 2 segundos
- Muestra nombre, foto y estado (puntual/tarde) al registrar
- Registra solo una vez por persona por día
- Permite cambiar entre cámara frontal y trasera
- Funciona sin internet (modo offline)

---

HU-02 — Notificación de asistencia (Padre)
Como padre de familia, quiero recibir una notificación en mi
celular cuando mi hijo llegue al colegio para saber si llegó
puntual o tarde.

Criterios de aceptación:
- La notificación llega en menos de 30 segundos del registro
- Indica nombre, hora de llegada y estado (puntual/tarde)
- Si llega tarde, indica cuántos minutos de retraso
- Funciona en segundo plano sin abrir la app

---

HU-03 — Reporte por rango de fechas (Administrador)
Como administrador, quiero generar un reporte de asistencia de
un rango de fechas para exportarlo a Excel y entregarlo a
dirección.

Criterios de aceptación:
- Permite filtrar por grado, tipo de persona y rango de fechas
- Muestra asistencias y faltas diferenciadas
- Exporta a PDF y Excel con formato legible
- Incluye docentes y estudiantes

---

HU-04 — Enrollment de nuevo alumno (Administrador)
Como administrador, quiero registrar el rostro de un nuevo
alumno para que el sistema pueda reconocerlo desde el primer
día.

Criterios de aceptación:
- Captura múltiples fotos del rostro
- Confirma el enrollment exitoso
- El alumno es reconocible en el siguiente reconocimiento
- Permite enrollment desde cámara web o imagen cargada

---

HU-05 — Sync offline (Operador móvil)
Como operador con app móvil, quiero registrar asistencia aunque
no haya internet, y que se sincronice sola cuando vuelva la
señal.

Criterios de aceptación:
- El app guarda registros localmente sin internet
- Al recuperar conexión, sincroniza automáticamente
- Los registros offline calculan tardanza correctamente
- No duplica registros ya existentes en el servidor

---

HU-06 — Historial de asistencia (Administrador)
Como administrador, quiero ver el historial completo de
asistencias del día con fotos y poder filtrar por nombre o
grado.

Criterios de aceptación:
- Muestra foto del rostro capturado y frame panorámico
- Filtros por nombre, grado, turno y estado de tardanza
- Actualización en tiempo real sin recargar la página
- Permite eliminar registros erróneos (super admin)


1.6 Criterios de Aceptación Generales
----------------------------------------------------------------

- El sistema reconoce correctamente a personas registradas con

3.1 Descripción de la Arquitectura
----------------------------------------------------------------

QhawAI utiliza una arquitectura de TRES CAPAS CON MICROSERVICIO
DE IA, compuesta por:

1. Capa de presentación
   - Aplicación web (Laravel Blade + Tailwind CSS + JavaScript)
   - Aplicación móvil Android (QhawAI APK)

2. Capa de lógica de negocio
   - Backend Laravel 13.5 (PHP)
   - Gestiona: autenticación, reglas de negocio,
     notificaciones, persistencia y reportes

3. Microservicio de IA
   - API REST en Python con FastAPI + Uvicorn
   - Contenedor Docker con Nginx (puerto 80 interno)
   - Ejecuta: YOLO-Face (detección) + InsightFace/ArcFace
     (reconocimiento)
   - Guarda: fotos de rostro y frames en sistema de archivos

4. Capa de datos
   - Base de datos MySQL (datos relacionales)
   - Sistema de archivos local (imágenes de asistencia)
   - JSON en disco (configuración de turnos para Python)

3.4 Registro de Decisiones Arquitectónicas
----------------------------------------------------------------

DA-01 — Python separado de Laravel para reconocimiento facial
Decisión: El reconocimiento facial se implementó como
microservicio independiente en Python, no dentro de Laravel.
Motivo: Los modelos de IA (InsightFace, YOLO) son bibliotecas
Python sin equivalente maduro en PHP. Esta separación permite
actualizar el modelo de IA sin modificar el backend web.
Alternativa descartada: Ejecutar Python como proceso hijo desde
PHP (mayor acoplamiento, difícil de escalar).

DA-02 — MySQL como única fuente de verdad para asistencias
Decisión: Laravel escribe todas las asistencias en MySQL.
Python solo reconoce y devuelve resultados, no persiste datos
relacionales.
Motivo: Centralizar en Laravel permite usar Eloquent ORM,
migraciones y el sistema de notificaciones sin duplicar lógica.

DA-03 — Imágenes almacenadas en el servidor Python
Decisión: Las fotos de rostro y frames panorámicos se guardan
en el sistema de archivos del servidor Python y se sirven como
archivos estáticos.
Motivo: Python tiene acceso directo al frame de la cámara en
el momento del reconocimiento. Evita transferir imágenes
innecesariamente de Python a Laravel.

DA-04 — Proxy de imágenes en Laravel (/img-proxy)
Decisión: Laravel implementa una ruta /img-proxy que re-sirve
imágenes HTTP del servidor Python como HTTPS.
Motivo: El servidor Python puede correr en HTTP internamente.
Sin el proxy, el navegador bloquearía las imágenes por política
de Mixed Content en páginas HTTPS.

DA-05 — firstOrCreate en sync offline
Decisión: Al sincronizar registros offline, se usa firstOrCreate
en lugar de updateOrCreate.
Motivo: Si la persona ya fue registrada online ese día (con
foto facial), el registro offline no debe sobrescribir el
registro original.

DA-06 — Frame panorámico guardado a baja calidad
Decisión: Los frames panorámicos se guardan a 640px de ancho
y 45% de calidad JPEG.
Motivo: Un frame a calidad completa pesa ~200KB. A baja calidad
pesa ~45KB. Para 500 reconocimientos/día en 200 días escolares,
esto reduce el almacenamiento de ~20GB a ~4.5GB por año.

Anexo D — Glosario
----------------------------------------------------------------

ArcFace: Modelo de reconocimiento facial basado en redes
         neuronales profundas que genera embeddings únicos
         por persona.

Embedding: Vector numérico de alta dimensión que representa
           el rostro de una persona de forma única.

Enrollment: Proceso de registrar el rostro de una persona en
            el sistema para su posterior reconocimiento.

FCM: Firebase Cloud Messaging. Servicio de Google para enviar
     notificaciones push a dispositivos Android.

InsightFace: Biblioteca Python open-source para detección y
             reconocimiento facial de alta precisión.

ONNX: Open Neural Network Exchange. Formato estándar para
      exportar e importar modelos de IA entre frameworks.

Turno: División del día escolar. Turno mañana (7:00-13:00)
       y turno tarde (13:00-18:00).

YOLO-Face: Variante del modelo YOLO optimizada para la
           detección de rostros en imágenes y video.

Snapshot: Copia de los datos en un momento específico.
          En asistencia, el nombre y grado guardados al
          momento del registro, independientes de cambios
          futuros.
