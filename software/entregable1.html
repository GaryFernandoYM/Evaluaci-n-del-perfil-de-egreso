Entregable 1 · CE021 — Ingeniería de Software

Requerimientos y Diseño del Sistema

Especificación completa de requerimientos funcionales, no funcionales, historias de usuario, diseño arquitectónico y diagramas UML del sistema QhawAI.

1.1 Requerimientos Funcionales

RF-01
Registro de asistencia por reconocimiento facial
El sistema debe capturar imágenes desde la cámara web o móvil, procesarlas mediante el modelo de reconocimiento facial y registrar automáticamente la asistencia, almacenando fecha, hora, turno y nivel de confianza.
RF-02
Detección de tardanza
El sistema debe comparar la hora de llegada con el horario de entrada configurado por turno (mañana/tarde) más la tolerancia en minutos, marcando el registro como tardanza y calculando los minutos de retraso.
RF-03
Enrollment (registro de rostros)
El sistema debe permitir registrar nuevos usuarios capturando múltiples imágenes de su rostro para construir su perfil biométrico en la base de datos del modelo de IA.
RF-04
Modo offline con sincronización
La aplicación móvil debe registrar asistencias localmente cuando no hay conexión a internet y sincronizarlas automáticamente con el servidor al recuperar la conexión.
RF-05
Notificaciones a padres
El sistema debe enviar notificaciones push (FCM) a los padres de familia registrados cuando su hijo registra asistencia, indicando si llegó puntual o tarde y la hora exacta.
RF-06
Gestión de usuarios y roles
El sistema debe gestionar tres roles: administrador (acceso total), operador (toma de asistencia) y padre de familia (consulta de asistencia de sus hijos vinculados).
RF-07
Reportes de asistencia
El sistema debe generar reportes de asistencia filtrados por día, persona o rango de fechas, mostrando asistencias y faltas, exportables en formato PDF y Excel.
RF-08
Historial de asistencia
El sistema debe mostrar el historial con foto del rostro capturado, nombre, grado, hora, turno y estado de puntualidad, con filtros por nombre, tipo, grado, turno y fecha.
RF-09
Gestión de grados y secciones
El sistema debe permitir crear, editar y eliminar grados, secciones y áreas, con configuración individual de horario de entrada y tolerancia de tardanza por grado.
RF-10
Cambio de cámara en dispositivos móviles
La aplicación debe detectar si el dispositivo tiene más de una cámara y permitir al operador cambiar entre cámara frontal y trasera durante la toma de asistencia.
RF-11
Registro de ubicación GPS
El sistema debe registrar las coordenadas GPS del dispositivo al momento de cada registro de asistencia.
RF-12
Evidencia fotográfica
El sistema debe guardar automáticamente el recorte del rostro reconocido y el frame panorámico de la cámara (en baja resolución) como evidencia de cada registro de asistencia.
RF-13
Panel de administración
El sistema debe proveer un panel web para gestionar personas, usuarios, grados, configuración de horarios y visualizar reportes de actividad de operadores.
RF-14
Sincronización de configuración con la API Python
El sistema debe sincronizar la configuración de turnos (hora de entrada, tolerancia, corte) desde el panel web hacia la API de reconocimiento facial automáticamente al guardar cambios.

1.2 Requerimientos No Funcionales

RNF-01 — Rendimiento
Respuesta de reconocimiento en <2 segundos bajo condiciones normales de red.
RNF-02 — Precisión
Precisión mínima del 90% en condiciones de iluminación adecuada. Umbral de confianza configurable.
RNF-03 — Disponibilidad
Sistema web disponible el 95% del tiempo durante horario escolar (L–V, 6:00–14:00).
RNF-04 — Seguridad
Todas las rutas con autenticación. Comunicación por API Key. Contraseñas con hash bcrypt.
RNF-05 — Escalabilidad
Soporte de hasta 1000 personas registradas sin degradación significativa del rendimiento.
RNF-06 — Usabilidad
Interfaz operable por usuario sin capacitación técnica en <5 minutos de uso.
RNF-07 — Compatibilidad
App Android compatible con Android 8.0 (API 26) o superior.
RNF-08 — Almacenamiento
Fotos panorámicas: máx 640px, calidad 45% JPEG. Recortes de rostro: calidad 88%.
RNF-09 — Privacidad biométrica
Embeddings faciales almacenados únicamente en el servidor institucional. No se comparten con terceros.
RNF-10 — Responsive
Interfaz web adaptable a escritorio (1024px+) y dispositivos móviles (320px+).

1.3 Reglas de Negocio

RN-01
Una persona solo puede tener un registro de asistencia por día. Los intentos duplicados se ignoran.
RN-02
El sistema no registra asistencia fuera del horario de corte configurado por turno. El intento se marca como "fuera de hora".
RN-03
Un registro de asistencia offline no puede sobrescribir un registro previo del mismo día. El primero en llegar tiene prioridad.
RN-04
Las notificaciones a padres solo se envían para estudiantes, no para docentes.
RN-05
El porcentaje de confianza mínimo para registrar asistencia es configurable. Por defecto es 55%.
RN-06
Un día es considerado "día escolar válido" para el cálculo de faltas si asistió al menos el 10% del total de personas registradas.
RN-07
Los reportes de faltas incluyen a todas las personas (estudiantes y docentes) que no tienen registro en un día escolar válido.

1.5 Historias de Usuario

HU-01 — Toma de asistencia (Operador)
"Quiero apuntar la cámara hacia los alumnos que ingresan para que el sistema registre automáticamente su asistencia sin que tengan que hacer nada."
Criterios de aceptación:
  • El sistema detecta el rostro en menos de 2 segundos
  • Muestra nombre, foto y estado (puntual/tarde) al registrar
  • Registra solo una vez por persona por día
  • Permite cambiar entre cámara frontal y trasera
  • Funciona sin internet (modo offline)
HU-02 — Notificación de asistencia (Padre)
"Quiero recibir una notificación en mi celular cuando mi hijo llegue al colegio para saber si llegó puntual o tarde."
Criterios de aceptación:
  • La notificación llega en menos de 30 segundos del registro
  • Indica nombre, hora de llegada y estado (puntual/tarde)
  • Si llega tarde, indica cuántos minutos de retraso
  • Funciona en segundo plano sin abrir la app
HU-03 — Reporte por rango de fechas (Administrador)
"Quiero generar un reporte de asistencia de un rango de fechas para exportarlo a Excel y entregarlo a dirección."
Criterios de aceptación:
  • Permite filtrar por grado, tipo de persona y rango de fechas
  • Muestra asistencias y faltas diferenciadas
  • Exporta a PDF y Excel con formato legible
  • Incluye docentes y estudiantes
HU-04 — Enrollment de nuevo alumno (Administrador)
"Quiero registrar el rostro de un nuevo alumno para que el sistema pueda reconocerlo desde el primer día."
  • Captura múltiples fotos del rostro
  • Confirma el enrollment exitoso
  • El alumno es reconocible en el siguiente reconocimiento
  • Permite enrollment desde cámara web o imagen cargada
HU-05 — Sync offline (Operador móvil)
"Quiero registrar asistencia aunque no haya internet, y que se sincronice sola cuando vuelva la señal."
  • El app guarda registros localmente sin internet
  • Al recuperar conexión, sincroniza automáticamente
  • Los registros offline calculan tardanza correctamente
  • No duplica registros ya existentes en el servidor
HU-06 — Historial de asistencia (Administrador)
"Quiero ver el historial completo de asistencias del día con fotos y poder filtrar por nombre o grado."
  • Muestra foto del rostro capturado y frame panorámico
  • Filtros por nombre, grado, turno y estado de tardanza
  • Actualización en tiempo real sin recargar la página
  • Permite eliminar registros erróneos (super admin)

3. Diseño Arquitectónico

🏗️
QhawAI utiliza una arquitectura de tres capas con microservicio de IA, con separación total entre el backend web (Laravel/PHP) y el motor de reconocimiento facial (Python).
Capa 1 — Presentación
Aplicación web (Laravel Blade + Tailwind CSS + JavaScript vanilla) y aplicación móvil Android.
Capa 2 — Lógica de Negocio
Backend Laravel 13.5 (PHP). Gestiona autenticación, reglas de negocio, notificaciones, persistencia y reportes.
Microservicio IA
FastAPI en contenedor Docker (Nginx → Uvicorn). Toda la IA corre en Python: YOLO-Face detecta rostros, InsightFace buffalo_l genera embeddings ArcFace (512-d) para reconocimiento. Laravel solo consume el resultado vía HTTP en http://localhost:80.
Capa 3 — Datos
Base de datos MySQL (datos relacionales), sistema de archivos local (imágenes), JSON en disco (config de turnos para Python).

Diagrama de Arquitectura

Diagrama de arquitectura del sistema QhawAI

Decisiones Arquitectónicas (ADR)

IDDecisiónMotivo
DA-01 Python separado de Laravel Los modelos de IA (InsightFace, YOLO) son bibliotecas Python sin equivalente maduro en PHP. Permite actualizar el modelo sin modificar el backend web.
DA-02 MySQL como única fuente de verdad Laravel escribe todas las asistencias en MySQL. Python solo reconoce y devuelve resultados, sin persistir datos relacionales.
DA-03 Imágenes almacenadas en servidor Python Python tiene acceso directo al frame de la cámara. Evita transferir imágenes innecesariamente de Python a Laravel.
DA-04 Proxy de imágenes /img-proxy El servidor Python puede correr en HTTP internamente. Sin el proxy, el navegador bloquearía imágenes por Mixed Content en páginas HTTPS.
DA-05 firstOrCreate en sync offline Si la persona ya fue registrada online ese día (con foto facial), el registro offline no debe sobrescribir el registro original.
DA-06 Frame panorámico a baja calidad A calidad completa: ~200KB/frame. A 640px + 45% JPEG: ~45KB. Esto reduce el almacenamiento de ~20GB a ~4.5GB por año con 500 reconocimientos/día.

4.2 Diagrama de Secuencia — Reconocimiento Facial

Flujo completo desde que el operador inicia la cámara hasta que la asistencia queda registrada en MySQL y el padre recibe la notificación push.

  1. 1Operador presiona "Iniciar" → Navegador solicita acceso a la cámara con getUserMedia()
  2. 2JavaScript captura un frame cada 200 ms como base64 JPEG (75% calidad)
  3. 3POST /attendance/recognize → Laravel reenvía a FastAPI POST /api/recognize (Docker :80)
  4. 4FastAPI: YOLO-Face detecta rostros → ArcFace genera embeddings → compara con base registrada
  5. 5FastAPI retorna {faces, is_late, minutos_tarde, paths}; guarda fotos en background thread
  6. 6Laravel: INSERT asistencia en MySQL → notifyParents() → envía push FCM al padre
  7. 7Navegador recibe JSON → actualiza overlay en canvas y paneles laterales en tiempo real
Diagrama de secuencia — flujo completo del reconocimiento facial y registro de asistencia

4.3 Diagrama de Estados — Registro de Asistencia

Ciclo de vida de un registro de asistencia desde que se detecta un rostro hasta el estado final almacenado en la base de datos.

Flujo normal
Inactivo → cámara activa
→ Procesando (rostro detectado)
→ Verificar duplicado (dni+fecha)
→ Verificar horario de corte
→ Calcular tardanza
→ Guardado en BD
→ Notificación push enviada
Estados de salida temprana
Rostro no identificado → Desconocido
Ya registrado hoy → Duplicado — Ignorado
Fuera de horario de corte → Fuera de Tiempo
Confianza < umbral → Rechazado
